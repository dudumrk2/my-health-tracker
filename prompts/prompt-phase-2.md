קרא קודם את `CLAUDE.md` ו-`HLD-health-tracker.md` (במיוחד סעיף Function Contracts → `analyzeMeal`, וסעיף Prompt Contracts → Contract A). פייז 1 כבר קיים — בנה עליו, אל תשבור אותו.

**המשימה:** ממש את פייז 2 — רישום ארוחות עם ניתוח AI.

**צד הלקוח (Android):**
1. מסך רישום ארוחה עם שתי דרכי קלט: טקסט חופשי, וצילום/בחירת תמונה.
2. שליחת הקלט ל-Cloud Function `analyzeMeal` (HTTPS Callable, מאומת Auth + App Check).
3. הצגת התוצאה המובנית (פריטים + ערכים תזונתיים) למשתמש.
4. **אפשר עריכה ידנית של התוצאה לפני שמירה** (מודלי vision טועים בכמויות — חשוב).
5. שמירה סופית ל-`users/{uid}/meals/{mealId}`.
6. **אל תשמור את התמונה** בשום מקום — רק התוצאה המנותחת.
7. **יומן מים:** כפתור "+ מים" מהיר בטאב אוכל (לחיצה אחת = כוס). שמירה/עדכון `users/{uid}/water/{date}` (שדה `cups`, idempotent — הגדלה ב-1).

**צד השרת (Cloud Function `analyzeMeal`):**
1. קלט/פלט לפי החוזה ב-HLD (inputType, text/imageBase64, date → items[] + totals).
2. קריאה ל-Gemini via Vertex AI (vision לתמונה, טקסט לתיאור). המפתח/הגישה בצד השרת בלבד.
3. Gemini מחזיר JSON בלבד לפי הסכמה; דגל `lowConfidence` כשלא ודאי; ללא טקסט עוטף.
4. Resilience: timeout 15s ל-Vertex, retry max 2 על 429/503/timeout בלבד, מעטפת שגיאה אחידה.
5. בכשל ניתוח → שגיאה ידידותית שמאפשרת ללקוח ליפול חזרה להזנה ידנית.
6. הפעל Firebase App Check; דחה קריאות ללא טוקן תקף.

**טסטים (חובה):**
- שרת: פלט JSON תקין/לא-תקין, טיפול בכשל Gemini (mock), אכיפת מעטפת שגיאה, מקרה "אין מזון בתמונה" → `items: []`.
- לקוח: זרימת קלט→תוצאה→עריכה→שמירה; מצב שגיאה→הזנה ידנית.

**אסור:** אל תיגע בתובנות ה-AI / Cloud Scheduler (פייז 3). אל תשמור תמונות. אל תכניס גישת Gemini לצד הלקוח.

**עצור ובקש ממני** לפני: פריסת Functions (`firebase deploy --only functions`), הפעלת Vertex AI API, הגדרת App Check. אלה פעולות ידניות שלי.
